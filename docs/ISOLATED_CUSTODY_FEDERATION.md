# 🔐 Изолированная кастодия — федеративный пирамидный турнир (инструкция администратора)

Инструкция по проведению федеративного пирамидного турнира, где **бай-ин каждого игрока принимается он-чейн
на отдельный, выделенный под этот турнир и этого игрока кошелёк Solana (USDT)**. Все операции требуют роли
**`ADMIN`**.

> База API: `/api/v1` (контекст-путь `/api` + префикс `/v1`). Веб-интерфейс: страница **`/admin/federation`**.
> Все админ-вызовы идут с `Authorization: Bearer <JWT>`.
> Пути ниже даны относительно `/api/v1/admin/pyramid-federations`.

---

## 0. Главный принцип безопасности

**Приватные ключи и seed НИКОГДА не попадают на сервер и в браузер.** Сервер хранит только публичные адреса
(watch-only), индексы деривации и метаданные. Любая подпись (пред-создание/закрытие ATA, рефанд) делается на
**офлайн-машине** (air-gapped), которая держит seed. Веб-консоль только собирает *неподписанную* транзакцию,
показывает её, принимает уже подписанную и шлёт в сеть. Если что-то предлагает «загрузить seed/приватный ключ»
на сервер или в браузер — так делать нельзя.

Дополнительно: **изоляция работает «в покое»** (деньги каждого игрока лежат на своём адресе). При выплате призов
on-chain она бы разрушилась (связываемость), поэтому он-чейн-сеттлинг призов в этой модели **не делается** —
призы начисляются off-chain в обычный кошелёк победителя.

## 2. Жизненный цикл — кратко

```
создать турнир (изолированный)
   → сгенерировать ключи офлайн → импортировать кошельки (адреса)
   → пред-создать ATA (офлайн-подпись оператора)
   → игроки регистрируются (каждому выдаётся его адрес) и платят бай-ин он-чейн
   → reconcile депозитов (вручную или авто-поллер) → оплаченные сажаются в шарды
   → release no-show (вручную или авто) — неоплаченные брони освобождаются
   → промоут шардов → финал → распределение призов (off-chain)
   → при отмене/разрегистрации: рефанд полного бай-ина (с согласия админа)
   → закрыть ATA → вернуть rent оператору
```

---

## 3. Создание турнира

**Веб:** `/admin/federation` → форма Create → отметить **«Isolated custody»** (бай-ин форсится в `USDT_SOL`,
сумма обязательна).

**API:** `POST /` (тело `CreateFederationRequest`):
```json
{ "name": "...", "startingPlayers": 1000, "shardSize": 100,
  "buyInAmount": 5, "isolatedWalletsEnabled": true }
```
Изолированная кастодия требует ненулевой бай-ин и ассет `USDT_SOL` (валидируется на бэкенде).

## 4. Генерация и импорт выделенных кошельков

### 4.1 Офлайн-генерация (на air-gapped машине)
Инструмент — main-класс `com.truholdem.tools.OfflineDepositPoolGenerator` (в test-исходниках). Режим
`--federation-id=…` детерминированно выводит кошельки **по абсолютному индексу** и пишет их чанками, не держа
всё поле в памяти. Аргументы в формате **`--ключ=значение`**:

| Аргумент | Значение | По умолчанию |
|---|---|---|
| `--federation-id=<UUID>` | id турнира (берётся из ответа на создание, §3) | — (обязателен) |
| `--count=<N>` | сколько кошельков сгенерировать | `1000` |
| `--chunk=<размер>` | размер чанк-файла | `10000` |
| `--mint=<base58>` | mint USDT в Solana | mainnet USDT |
| `--out-dir=<папка>` | куда писать файлы | `.` |
| `--seed-hex=<64 hex>` | мастер-seed (если задан — детерминированно; иначе случайный 32-байтовый) | случайный |

Запуск (нет exec-плагина — через classpath; PowerShell/Windows):
```powershell
cd backend
.\mvnw -q test-compile
.\mvnw -q org.apache.maven.plugins:maven-dependency-plugin:build-classpath "-Dmdep.outputFile=cp.txt"
java -cp "target/test-classes;target/classes;$(Get-Content cp.txt)" `
  com.truholdem.tools.OfflineDepositPoolGenerator `
  --federation-id=<UUID> --count=100000 --chunk=10000 --out-dir=out
```

На выходе в `--out-dir`:
- `fedwallets-import-00000.json`, `…-00001.json`, … — **публичные** записи
  `{ "wallets": [ {derivationIndex, ownerPubkey, address} ] }`, готовые к импорту (§4.2);
- `fedwallets-secret.txt` — `seedHex` + `federationId` + `mint`. **Остаётся ТОЛЬКО на офлайн-машине** (по нему
  позже переподписываются ATA/рефанды через `federationWalletSeed(seed, fedId, index)`). На сервер не
  загружается, в репозиторий не коммитится.

> Приватные ключи как таковые на диск не пишутся: хранится мастер-seed, а ключ кошелька #i выводится из
> `(seed, federationId, i)` офлайн по требованию. Достаточно сохранить `fedwallets-secret.txt`.

Масштаб: для 1M кошельков генерируйте чанками (напр. по 10 000) — память ограничена одним чанком.

### 4.2 Импорт публичных адресов на сервер
**Веб:** консоль изолированной кастодии → «1 · Import dedicated wallets» → вставить содержимое чанк-файла.

**API:** `POST /{id}/import-wallets` (тело — `{ wallets: [...] }`).
Импорт **идемпотентен**: повторный импорт того же чанка добавит `0` (дедуп по адресу + bulk-вставка), поэтому
большое поле грузится чанк-за-чанком безопасно.

## 5. Пред-создание ATA (обязательно до приёма депозитов)

Биржевой вывод шлёт «голый» SPL-перевод и **не создаёт** ATA получателя. Поэтому ATA каждого выделенного
кошелька надо создать **до** публикации адреса игроку — иначе бай-ин не дойдёт. Создаёт оператор (платит rent
~0.00204 SOL/ATA; rent возвратный — см. §9). Делайте ATA для FREE-буфера **с запасом** перед регистрациями.

Поток (офлайн-подпись, **один** подписант = оператор):
| Шаг | Веб | API |
|---|---|---|
| 1. Собрать пакет | «Build create batch» (limit) | `POST /{id}/ata/create/unsigned?limit=N` → `messageBase64`, `walletIds`, `signers` |
| 2. Подписать | — (офлайн) | подписать `messageBase64` ключом оператора, собрать tx |
| 3. Отправить | «Broadcast» | `POST /{id}/ata/broadcast` `{signedTx}` → `{signature}` |
| 4. Подтвердить | «Confirm created» (signature + walletIds) | `POST /{id}/ata/create/confirm` `{signature, walletIds}` → `{provisioned}` |

Размер пакета ограничен `…-ata-batch-size` (по умолчанию 6) — чтобы tx укладывалась в лимиты размера/compute.

## 6. Приём депозитов: reconcile и no-show

Игрок регистрируется (`registerIsolated`) и получает **свой** адрес (ATA). Регистрация **посажена в шард только
после подтверждённого он-чейн депозита** ≥ бай-ин.

### 6.1 Reconcile (зачесть оплаченных)
**Веб:** «Reconcile deposits». **API:** `POST /{id}/reconcile-deposits` → `{seated}`.
Балансы читаются пачками по 100 (`getMultipleAccounts`) — поле из N кошельков стоит `ceil(N/100)` RPC-вызовов.

**Авто-режим (рекомендуется):** включите `…-deposit-poll-enabled=true` — фоновый поллер сам зовёт reconcile для
всех `REGISTERING` изолированных федераций каждые `…-poll-interval-ms`. Идемпотентно, безопасно в кластере.

### 6.2 Release no-show (освободить неоплаченные брони)
Если игрок взял адрес, но не заплатил за окно `…-deposit-window-minutes` — его бронь можно вернуть: кошелёк уходит
в `FREE` (с уже готовым ATA, переиспользуем), pending-регистрация **удаляется**.

**Веб:** «Release no-shows». **API:** `POST /{id}/release-no-shows` → `{released}`.

**Авто-режим:** `…-auto-release-enabled=true` — тот же поллер после reconcile сам освобождает no-show.
> ⚠️ Это **отдельный** opt-in (по умолчанию off), потому что он **удаляет** pending-регистрации. Порядок жёсткий:
> сначала reconcile, потом release — запоздавший, но валидный депозит будет зачтён, а не выброшен. Окно делайте
> с запасом (биржевой вывод идёт минутами).

## 7. Проведение турнира

Те же действия, что у обычной федерации (доступны на `/admin/federation` и через API):
`promote` (материализовать готовые шарды) · `drain-shards` / `run-final` (симуляция/боты) · `schedule-final`
(назначить финал + уведомить финалистов) · `start-final` · `distribute` / `prize-config`. Призы начисляются
**off-chain** в кошельки победителей.

---

## 8. Рефанды (с согласия администратора)

Если турнир отменён / недонабор / игрок разрегистрировался — профинансированный бай-ин возвращается. **Игрок
получает полный бай-ин** (net = gross, комиссия не удерживается — сетевую SOL-комиссию поглощает оператор). Это
держит книги честными и опустошает ATA, чтобы его потом закрыть (§9).

Подпись офлайн, **два** подписанта: оператор (fee-payer) + владелец выделенного кошелька (authority).
| Шаг | Веб | API |
|---|---|---|
| Запросить (1 игрок) | «Request refund» (playerId) | `POST /{id}/players/{playerId}/refund` |
| Запросить все (отмена) | «Request all (cancelled)» | `POST /{id}/refunds/request` → `{requested}` |
| **Одобрить** (адрес) | «Approve» (refundId + адрес) | `POST /refunds/{refundId}/approve` `{toAddress}` |
| Отклонить | «Reject» | `POST /refunds/{refundId}/reject` `{reason}` |
| Собрать unsigned | «Build unsigned» | `GET /refunds/{refundId}/unsigned` |
| Отправить | «Broadcast refund» | `POST /refunds/{refundId}/broadcast` `{signedTx}` |
| Подтвердить | «Reconcile» | `POST /refunds/{refundId}/reconcile` → кошелёк `REFUNDED` |

Ничего не подписывается, пока модератор не одобрит рефанд и не укажет адрес назначения. Адрес проверяется
(base58). Дубли запроса на один кошелёк идемпотентны.

---

## 9. Закрытие ATA — возврат rent

После завершения (кошельки `REFUNDED`/пустые) их ATA можно закрыть и **вернуть rent оператору** (~0.00204 SOL за
каждый). Закрытие требует **пустого** ATA — благодаря полному рефанду (§8) пыли не остаётся, закрытие проходит
напрямую. Подпись офлайн: оператор (fee-payer) + владелец каждого кошелька.

| Шаг | Веб | API |
|---|---|---|
| Собрать пакет | «Build close batch» (walletIds) | `POST /{id}/ata/close/unsigned` `{walletIds}` |
| Отправить | «Broadcast» | `POST /{id}/ata/broadcast` `{signedTx}` |
| Подтвердить | «Confirm closed» | `POST /{id}/ata/close/confirm` `{signature, walletIds}` → `{closed}` |

## 10. Дашборд пула

**Веб:** сетка статистики вверху консоли (авто-обновляется после каждой операции).
**API:** `GET /{id}/wallet-stats` → `FederationWalletStatsResponse`:

| Поле | Значение |
|---|---|
| `total` | всего импортировано кошельков |
| `free` | свободный буфер (ещё не выдан) |
| `assigned` | выдан, **ждёт он-чейн депозит** |
| `funded` | депозит подтверждён (игрок посажен) |
| `refunded` | бай-ин возвращён |
| `ataProvisioned` | у скольких ATA уже пред-создан |
| `fundedAmount` | суммарный собранный бай-ин он-чейн |

Контролируйте, чтобы `free` (с готовыми `ataProvisioned`) держался **с запасом впереди** регистраций.

## 11. Типовой чек-лист запуска

1. Включить флаги из §1 (`payments`, `sol-rpc`, `federated-pyramid`, `federated-isolated-wallets`); задать
   `sol-from-address`, `sol-usdt-mint`, `sol-rpc-url`.
2. Создать турнир (изолированный), §3.
3. Офлайн сгенерировать N+запас ключей; импортировать чанки, §4.
4. Пред-создать ATA для всего импортированного буфера, §5.
5. Включить авто-reconcile (`…-deposit-poll-enabled`) и, при желании, авто-release
   (`…-auto-release-enabled`), §6. Открыть регистрацию.
6. Следить по дашборду (§10); по окончании окна — no-show освобождаются (авто или вручную).
7. Промоут → финал → распределение призов, §7.
8. На отмену/разрегистрацию — рефанды (§8), затем закрыть ATA и вернуть rent (§9).
