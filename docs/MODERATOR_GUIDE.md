# 🛡️ Руководство модератора — крипто-кошелёк TruHoldem

Инструкция для модератора/администратора по проверке KYC, одобрению выводов, офлайн-подписи и управлению
пулом депозитных адресов. Все операции требуют роли **`ADMIN`**.

> База API: `/api/v1` (контекст-путь `/api` + префикс версии `/v1`). Веб-интерфейс: страницы `/admin/...`.
> Все админ-вызовы идут с заголовком `Authorization: Bearer <JWT>` (получается при входе админ-аккаунтом).
> Крипто-подсистема активна только при `app.payments.enabled=true`.

---

## 0. Главный принцип безопасности

**Приватные ключи и seed НИКОГДА не находятся на сервере.** Сервер хранит только публичные адреса
(watch-only) и метаданные. Подпись выводов делается на **офлайн-машине** (air-gapped), которая держит seed.
Если что-то предлагает «загрузить seed/приватный ключ на сервер» — это ошибка, так делать нельзя.

## 1. KYC: проверка верификации

Игрок загружает видео-верификацию (себя с паспортом) на `/kyc`; его KYC переходит в `PENDING`. Задача
модератора — посмотреть видео и вынести решение.

**Веб:** страница **`/admin/kyc`** — список ожидающих, просмотр видео, кнопки Approve / Reject / Erase.

**API:**
| Действие | Запрос |
|---|---|
| Список ожидающих | `GET /api/v1/admin/wallet/kyc/pending` |
| Скачать видео игрока | `GET /api/v1/admin/wallet/kyc/{userId}/document` |
| Решение | `POST /api/v1/admin/wallet/kyc/{userId}/decision` `{ "status": "VERIFIED" | "REJECTED", "note": "..." }` |
| Удалить медиа (GDPR) | `DELETE /api/v1/admin/wallet/kyc/{userId}/documents` |

**Что проверять на видео:** лицо человека и паспорт в кадре одновременно; данные паспорта читаемы; нет
признаков подделки/экрана. При сомнении — `REJECTED` с комментарием, игрок перезальёт.

**Важно:** только `VERIFIED` снимает гейт на вывод (если `kyc-required-for-withdrawal=true`). Видео —
чувствительный биометрический PII: оно шифруется на диске (AES-GCM) и доступно только ADMIN; по запросу
на удаление данных используйте `DELETE .../documents`. Старое медиа удаляется автоматически по сроку
хранения (`kyc-retention-days`, по умолчанию 30 дней).

## 2. Вывод средств: ручное одобрение

Включается флагом **`app.payments.withdrawal-approval-required=true`**. Тогда запрос на вывод **сразу
дебетует баланс** (резерв) и попадает в статус **`PENDING_APPROVAL`** — ждёт модератора.

**API:**
| Действие | Запрос |
|---|---|
| Список ожидающих | `GET /api/v1/admin/wallet/withdrawals` |
| Одобрить | `POST /api/v1/admin/wallet/withdrawals/{id}/approve` |
| Отклонить | `POST /api/v1/admin/wallet/withdrawals/{id}/reject` `{ "reason": "..." }` |

**Что делает одобрение/отклонение:**
- **Approve** → у обычного провайдера (gateway) сразу бродкаст (`BROADCAST`). У `offline-pool` остаётся
  `APPROVED` — деньги ещё не ушли, нужна офлайн-подпись (раздел 3).
- **Reject** → `REJECTED` + **возврат средств** игроку (реверс дебета). Деньги не уходят.

**Что проверять перед approve:** сумму и адрес назначения, историю игрока, статус KYC, нет ли подозрительных
паттернов (резкий вывод сразу после депозита, адрес из санкционных списков и т.п.). Действовать можно только
из `PENDING_APPROVAL` (повторное действие вернёт ошибку 409).

### Статусы вывода
```
PENDING_APPROVAL → APPROVED → BROADCAST → CONFIRMED
        │              │                     └─ (или FAILED → возврат средств)
        │              └─ (offline-pool: ждёт офлайн-подписи, см. §3)
        └─ REJECTED (модератор отклонил → возврат средств)
```

## 3. Офлайн-подпись вывода (air-gapped, для `offline-pool`)

Для самокастодиального пула одобренный вывод (`APPROVED`) надо подписать **офлайн** ключом из seed, затем
забродкастить через ноду. Сервер ноды не имеет — он отдаёт «намерение», подпись делается на изолированной
машине.

> **Перенос через воздушный зазор (текущий способ — USB/файл).** Между онлайн-машиной и офлайн-подписантом
> нет сети: данные переносятся вручную. Сейчас это **файл на флешке** — `unsigned`-JSON туда, подписанная
> `0x…`-строка обратно. Гигиена: отдельная флешка только для этого; ничего, кроме этих двух файлов, не
> переносить; на офлайн-машине перед подписью **глазами сверить `From address` и сумму**, которые печатает
> подписант (защита от подменённого намерения). Через зазор уходит только публичное намерение, обратно —
> только подпись; **seed/приватный ключ зазор не пересекают никогда.** (QR-перенос вместо USB — в планах,
> см. `TODO.md`; USB/файл останется задокументированным fallback.)

**Шаг 1 — экспорт намерения (онлайн):**
```
GET /api/v1/admin/wallet/withdrawals/{id}/unsigned
→ { withdrawalId, asset, network, toAddress, amount, createdAt }
```
Сохрани этот JSON и перенеси на офлайн-машину (флешка/QR).

**Шаг 2 — подпись (ОФЛАЙН-машина с seed):** запусти air-gapped подписант (Ethereum/ERC-20). Параметры цепи
(`nonce`, `gas-price`, `gas-limit`, `chain-id`) возьми с ноды/обозревателя; `index` — индекс адреса вывода
из seed.
```
java -cp <test-classpath> com.truholdem.tools.OfflineWithdrawalSigner \
  --seed-hex=<HEX_SEED> --index=<N> \
  --to=0x<recipient> --amount-wei=<int> \
  --nonce=<n> --gas-price-wei=<int> --gas-limit=21000 --chain-id=1 \
  [--token=0x<erc20-contract>]      # указать для USDT-ERC20 и т.п.
→ печатает подписанную raw-транзакцию (0x...)
```
Перенеси `0x...`-строку обратно на онлайн-машину. **Seed остаётся на офлайн-машине.**

**Шаг 3 — бродкаст (онлайн):** отправь raw-tx любой нодой:
```
eth_sendRawTransaction(0x...)  → получишь tx hash
```

**Шаг 4 — записать результат (онлайн):**
```
POST /api/v1/admin/wallet/withdrawals/{id}/broadcast   { "txId": "<tx-hash>" }
→ статус становится BROADCAST
```
Дальше провайдерский вебхук `withdrawal-status` переведёт его в `CONFIRMED` (или `FAILED` + возврат средств).

> BTC (PSBT) и TRON-подписанты — в планах; пока офлайн-подписант покрывает ETH и ERC-20 (USDT-ERC20).

## 4. Пул депозитных адресов

Депозитные адреса генерируются **офлайн** (приватные ключи не на сервере), на сервер импортируются только
публичные адреса; игрокам они выдаются по мере запроса.

**Мониторинг остатка:**
```
GET /api/v1/admin/wallet/deposit-pool/status   → free/assigned по каждому активу
```
Когда `free` для актива близко к нулю — пора досыпать пул.

**Генерация новой пачки (ОФЛАЙН-машина):**
```
java -cp <test-classpath> com.truholdem.tools.OfflineDepositPoolGenerator \
  --count=1000 --asset=ETH [--btc-style=p2pkh|bech32|taproot] --out-dir=.
→ private.json     (СЕКРЕТ: seed + ключи — хранить ОФЛАЙН, зашифровать)
→ addresses.json   (только публичные адреса — для импорта на сервер)
```

**Импорт публичных адресов (онлайн):**
```
POST /api/v1/admin/wallet/deposit-pool/import   <содержимое addresses.json>
→ { imported, skipped }   (повторный импорт дедуплицируется)
```

## 5. Чек-лист и правила

**Делать:**
- Проверять KYC-видео внимательно; решение фиксировать с комментарием.
- Перед approve вывода — сверять сумму/адрес/историю/KYC.
- Хранить `private.json` и seed **только офлайн**, в зашифрованном виде, с резервной копией (стальная
  пластина/зашифрованная флешка в сейфе).
- Для крупных сумм — холодное хранение + (по возможности) подпись двумя модераторами.

**Нельзя:**
- Загружать seed/приватные ключи на сервер или подключать офлайн-машину к интернету.
- Одобрять вывод без проверки адреса/истории.
- Отдавать KYC-видео кому-либо кроме уполномоченного модератора.

## 6. Справочник флагов (env / `application.properties`)

| Флаг | Назначение | По умолчанию |
|---|---|---|
| `app.payments.enabled` | Мастер-выключатель кошелька | `false` |
| `app.payments.provider` | `mock` / `gateway` / `eth-self-custody` / `offline-pool` | `mock` |
| `app.payments.kyc-required-for-withdrawal` | Требовать VERIFIED KYC для вывода | `true` |
| `app.payments.withdrawal-approval-required` | Ручное одобрение вывода модератором | `false` |
| `app.payments.min-confirmations` | Подтверждений до зачисления депозита | `1` |
| `app.payments.kyc-storage-dir` | Каталог KYC-видео (в кластере — общий volume) | — |
| `app.payments.kyc-encryption-key` | base64 AES-ключ шифрования видео at-rest | — (plaintext) |
| `app.payments.kyc-retention-days` | Авто-удаление KYC-медиа старше N дней (0 = никогда) | `30` |
